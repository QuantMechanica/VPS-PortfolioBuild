## Code Quality Findings

### PASS — No lookahead bias on signal bars (lines 91–119)

`HasLongBreakoutSignal()` and `HasShortBreakoutSignal()`:
- `iHighest/iLowest` start at shift=2 — window covers bars[-2] through bars[-21] (all completed)
- Signal bar is `iClose(_Symbol, _Period, 1)` — bar[-1], completed
- Bar 0 (current in-flight bar) never read

Boundary guard: if `hh_shift < 0` or `ll_shift < 0`, function returns false — safe for thin history windows.

### PASS — RISK_FIXED enforced (lines 14–16)

`RISK_PERCENT = 0.0`, `RISK_FIXED = 1000.0`. Consistent with DL-038 Rule 7 and setfile convention.

### PASS — IsNewBar guard (lines 39–46, 262)

All strategy logic gated on `IsNewBar()` in `OnTick()`. Correct pattern.

### PASS — Magic number (lines 33–37, registry)

Uses `QM_Magic(qm_ea_id, qm_magic_slot_offset)` directly. `magic_numbers.csv` has 36 entries for ea_id=1004 (slots 0–35, magic values 10040000–10040035). Registry consistent.

**Observation (non-blocking):** QM5_1003 uses `QM_FrameworkMagic()` (cached + symbol-validated), while QM5_1004 uses `QM_Magic()` directly. `QM_Magic()` is the base formula — result is identical for the same ea_id/slot. The difference is that `QM_FrameworkMagic()` enforces symbol-slot registry validation at init time. Direct `QM_Magic()` call skips runtime registry check. Not a correctness defect in backtesting (one symbol per terminal instance), but slightly weaker than the framework pattern.

### PASS — No standalone TP (lines 127, 203–204)

`req.tp = 0.0` on all entry paths. `CTrade.Buy/Sell` called with `tp=0.0`. Per card §4/§8.

### PASS — Position reversal logic (lines 207–227)

`ProcessSignalWithReversal()`: same-direction signal with open position → returns false. Opposite signal → `PositionClose` then `ExecuteEntrySignal`. Prevents double exposure.

### PASS — OnTick execution order (lines 251–270)

Order: KillSwitch → News → FridayClose → IsNewBar → ManagePosition → ExitSignal → EntrySignal. Standard V5 framework order.

### PASS — No unconstrained free parameters

Three strategy inputs: `breakout_lookback` (Card §6), `strategy_atr_period` (Card §6), `atr_stop_mult` (Card §4/§6). All referenced in card with WF defaults.

## NOTE — Dynamic stop widening from entry price (lines 156–176)

**Observation (non-blocking, verify against card §4):**

`Strategy_ManageOpenPosition()` recomputes stop each bar as `QM_StopRulesStopFromDistance(_Symbol, side, price_open, stop_distance)` — anchored to `price_open` (the position's entry price). If ATR increases post-entry, `stop_distance` grows, pushing the stop FURTHER from current price. This is the same dynamic widening pattern flagged in QM5_1003 pre-review.

Card §4 says "no trailing/partial logic; maintain protective stop only." If "protective" means a static stop from entry-bar ATR, the code is correct but widens unexpectedly on volatility spikes. Verify whether Card §4 explicitly specifies dynamic ATR adjustment or a static entry-bar stop.
