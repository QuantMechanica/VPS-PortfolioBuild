## Requested path checks

### (a) Configured same-day restart

Given a valid same-magic state file saved with offset `-1`, max-B/E `true`,
`halted=true`, halt reason `KS_DAILY_LOSS`, and baseline `94321.50`:

1. The new process starts with the default configuration. Restore reaches the
   configuration check before the day-key check and returns `FOREIGN_CONFIG`.
2. Init's line-523 guard therefore skips SaveState. The configured file survives
   unchanged.
3. `QM_KillSwitchSetDayAnchor(-1, true)` adopts the saved configuration and
   recomputes the day key before calling Restore.
4. Configuration and same-day key now match. Restore applies the original
   `94321.50` baseline and the daily-loss halt, then the setter persists that
   applied state.

Result: PASS — both halt and depletion baseline survive the documented
Init-plus-setter restart sequence.

### (b) Default-anchor restart

A same-day offset `0`, max-B/E `false` file matches Init's initial
configuration. Restore returns `APPLIED`, reinstates the original baseline and
daily-loss halt, and Init saves the applied state. The independent model and the
shipped `test_default_restart_still_restores_halt` both pass.

Result: PASS — existing default-anchor behavior is unregressed.

### (c) Genuinely stale foreign file

The preserve outcome cannot resurrect yesterday's halt after the configuration
is adopted on a later kill-switch day:

1. Init sees yesterday's offset `-1`/max-B-E `true` file while globals are still
   at defaults. It returns `FOREIGN_CONFIG` and temporarily preserves the file.
2. The setter adopts `-1`/`true` and computes today's configured day key.
3. The second restore now passes the configuration check, reaches line 228, and
   rejects `saved_day_key != g_qm_ks_day_key` with `NONE` before either the
   baseline or halt can be applied.
4. The unchanged setter path saves today's fresh anchor and `halted=false`,
   replacing the stale file.

The independent trace used yesterday `2026200`, today `2026201`, and fresh
anchor `100000.00`; it ended with day key `2026201`, anchor `100000.00`, and
`halted=false` both in memory and in the saved model file.

Result: PASS — preservation is only a bridge to the post-init configuration;
staleness is judged under that configuration before any old state is applied.

### (d) Python model versus MQH semantics

The behavioral model transcribes the source's relevant gates and mutations in
the same order: file presence, magic/positive-anchor validity, configuration,
day key, baseline application, daily-loss-only halt application, Init's
foreign-only save suppression, and the setter's restore-then-save sequence. Its
setter also preserves the source's active-halt day-key rebasing behavior.

The model's `_key(day, offset)` tuple is a symbolic configured-day key rather
than the MQH's literal `year * 1000 + day_of_year` integer. This is equivalent
for the covered lifecycle cases because day-key comparison occurs only after
offset/max-B-E equality, and the model's `day` denotes the already shifted
anchor day. It does not invert or bypass any source decision.

The shipped suite has a stale previous-day default-config test rather than a
dedicated stale-foreign-setter case. That is not a correctness gap in the fix:
the independent proof above walks the exact requested foreign/stale path through
the same model semantics and source order.

Result: PASS — the model matches the reviewed MQH semantics for every asserted
lifecycle outcome.
