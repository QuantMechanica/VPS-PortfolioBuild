## Task

Router task `f6769583-8cc0-4b91-b0e1-f2e641e6ca05`: add durable worker protection for EAs that write runaway tester journals, with specific incident context for `QM5_9917` and `QM5_1218`.

## Changes

- Added `RUN_JOURNAL_SIZE_CAP_BYTES = 2 GiB` to `tools/strategy_farm/terminal_worker.py`.
- During `_run_claimed_item`, the worker now polls the wrapper log path already stored in payload and recent MT5 tester journals under `D:/QM/mt5/<terminal>/Tester/logs`.
- The worker records `started_at_epoch` and ignores stale terminal journals older than the current spawn window, so an old large log does not poison a new claim.
- If the log exceeds the cap:
  - stop the child process tree,
  - stop the terminal slot via the existing release helper,
  - write `log_bomb_failure.json` under the work item report root,
  - set the work item to `failed` / `INFRA_FAIL`,
  - store payload fields `final_failure=LOG_BOMB`, `verdict_reason=INFRA_FAIL:LOG_BOMB`, and `log_bomb` size details,
  - mark the EA row in `framework/registry/ea_id_registry.csv` as `quarantined` when present.
- Added `test_run_claimed_item_log_bomb_fails_and_quarantines_ea` to `tools/strategy_farm/tests/test_terminal_worker_atomic_claim.py`; it now proves a growing terminal `Tester/logs/*.log` file trips the guard even when the wrapper log is small.
