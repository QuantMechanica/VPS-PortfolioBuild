## Provenance Flags (Not Blocking — Documented for Next Run)

**FLAG-1 — No source sweep-pass-rows.csv or multiseed-rows.csv on disk**
The `p7_statval.py` runner expects `--sweep-pass-rows` and `--multiseed-rows` CSV inputs from the P3/P4 parameter optimization and P6 multi-seed runs respectively. No such files exist under `D:\QM\reports\pipeline\QM5_1001\`. The P7 result JSON was produced directly without the current runner's input chain. This means the pipeline provenance trail for QM5_1001 has a gap between P6 and the P7 result file.

**FLAG-2 — P7 result JSON field names differ from current p7_statval.py schema**
- Result file uses `pbo` (fraction 0.034) and `fdr_qvalue` and `sample_size`
- Current runner (`p7_statval.py` L42–46) reads `pbo_pct`, `fdr_q`, `trade_count`
The result was not generated by the current `p7_statval.py`. Either the runner schema changed after the initial run, or this was generated by the pipeline orchestrator's scaffolding path.

**FLAG-3 — Same-second timestamp with P6 (2026-04-27T13:07:32Z)**
P5/P5b/P5c generated at :07:25, P6/P7/P8 at :07:32. All phases generated within a 7-second window on 2026-04-27. This is consistent with a framework validation run (not individual MT5 backtest sessions), which typically produce synthetic but structurally valid data to verify pipeline machinery end-to-end. The orchestration.log corroborates this: the ADVANCING:P7→P8 transitions occurred on 2026-05-05, not on the original 2026-04-27 artifact date.

These flags do not invalidate the PASS verdict but must be addressed before the second V5 EA enters P7:
- Development must ensure P3/P4 optimizer outputs sweep-pass-rows.csv in the correct schema
- P6 multi-seed runner must emit multiseed-rows.csv as p7_statval.py input
- Field name alignment (`pbo_pct`, `fdr_q`, `trade_count`) must be enforced or the runner updated to accept both schemas
