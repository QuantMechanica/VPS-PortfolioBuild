## Look-Ahead Analysis (Critical Path)

`HasLongBreakoutSignal()` uses:

```mql5
const int hh_shift = iHighest(_Symbol, _Period, MODE_HIGH, breakout_lookback, 2);
```

Start position = **2** (not 1). The lookback window spans bars `[2 .. 2+breakout_lookback-1]`, which **excludes bar[1]** (the trigger bar). Compare then:

```mql5
const double trigger_close = iClose(_Symbol, _Period, 1);   // bar[1] — closed
const double prior_high    = iHigh(_Symbol, _Period, hh_shift);  // hh_shift >= 2
```

Signal: `close[1] > max(high[2..lookback+1])`. Bar[0] (live bar) never touched.  
**No look-ahead. Confirmed clean.**

## Flag: Management Stop Widening (Medium)

`Strategy_ManageOpenPosition()` (lines 156-176):

```mql5
const double stop_distance = ResolveStopDistancePrice();   // current ATR * mult
const double sl = QM_StopRulesStopFromDistance(_Symbol, side, price_open, stop_distance);
g_trade.PositionModify(_Symbol, sl, 0.0);
```

Stop is recalculated **every new bar** as `ATR[1] * atr_stop_mult`, anchored to the original `price_open`. This means:

- **If ATR shrinks post-entry** → stop tightens toward entry (trailing-like). ✓  
- **If ATR expands post-entry** → stop widens away from entry, exposing **more capital than the initial risk sizing intended**.

The lot size set at entry is sized for `ATR-at-entry * 2.0`. If ATR doubles mid-trade, the effective risk per trade also doubles. Card §4 states "maintain protective stop only" — this comment in code (line 158) is consistent with the implementation being intentional, but the widening side-effect is worth a formal card reference check at P2.

**Assessment:** Not a code bug — the math is correct for both directions. But Development should confirm with the Card §4 spec that ATR-dynamic stop widening post-entry is deliberate. Flag for formal review.
