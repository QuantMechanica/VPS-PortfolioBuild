# Public Data Snapshots

JSON snapshot files exported hourly for quantmechanica.com.

Schema and snapshot contract: `docs/ops/OPTION_A_STRATEGY_FARM_RUNBOOK.md`.

## v1 Contract (PC6-01)

Every published snapshot file carries `"schema_version": 1` at the top level. Schemas pin
the version with `enum: [1]` and reject any other value. Schemas also set
`additionalProperties: false`, so silently widening the surface is rejected — adding a new
field is a breaking change that must bump the schema version.

Version-bump policy:

- v1 is frozen. Producers must emit `schema_version: 1` exactly.
- Adding, removing, or changing the meaning of any field requires a new schema file
  (e.g. `public-snapshot.schema.v2.json`), an `enum: [2]` pin, and a parallel publishing
  window so consumers can migrate.
- Renaming an enum value (e.g. `t6.status`) is breaking even if the field stays.

Stable identifiers are exposed via JSON Schema `$id`:

- `https://quantmechanica.com/schemas/public-snapshot/v1.json`
- `https://quantmechanica.com/schemas/process-roadmap/v1.json`
- `https://quantmechanica.com/schemas/strategy-archive/v1.json`
- `https://quantmechanica.com/schemas/company-operating-model/v1.json`

## Contracts

- `public-snapshot.schema.json` — top-level project state (phase, agents, pipeline counts, T6, expenses).
- `process-roadmap.schema.json` — list of public processes with status.
- `strategy-archive.schema.json` — strategy-archive items with visibility flag.
- `company-operating-model.schema.json` — operating-model menu/dashboard config.

## Runtime Files

Generated by `scripts/export_public_snapshot.ps1`:

- `public-snapshot.json`
- `process-roadmap.json`
- `strategy-archive.json`
- `company-operating-model.json`

Files are emitted as UTF-8. Consumers must accept an optional UTF-8 BOM (Windows
PowerShell `Set-Content -Encoding UTF8` writes one). Browsers and `JSON.parse` handle this
transparently; Python loaders should use `encoding='utf-8-sig'`.

## Validation

Run before commit / before publishing to confirm every snapshot conforms to its v1 schema
and that the negative fixtures still fail:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts\validate_public_snapshot.ps1
```

The validator iterates all four `(data, schema, negative-fixture)` triples and exits
non-zero on any failure. Negative fixtures live in
`scripts/tests/fixtures/public_snapshot_validation/` and exercise
`additionalProperties: false`, missing-required, and wrong-enum cases.

## Stale Data Behavior

- `company-operating-model.json` includes `updated_at` and `cache_ttl_minutes`.
- UI should render `As of <updated_at>` from `stale_data_behavior.ui_label_template`.
- If `now - updated_at > cache_ttl_minutes`, render a stale-warning state and keep last valid data visible.
- `public-snapshot.json.generated_at` older than 90 minutes triggers the dashboard stale-warning state.
